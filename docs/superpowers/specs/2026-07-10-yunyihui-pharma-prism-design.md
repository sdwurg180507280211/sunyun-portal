# 北京云医荟科技有限公司门户「医药棱镜」设计

日期：2026-07-10

状态：已完成视觉方向确认，等待书面规格复核

设计方向：医药棱镜 / Pharma Prism

## 1. 背景与目标

项目由“北京榫合云科技有限公司通用软件定制门户”转型为“北京云医荟科技有限公司医药行业数字化门户”。公司面向机构客户，核心客户为药品生产企业、医疗机构和医药商业公司，提供业务软件、数据平台、AI 工作流工具、系统集成和持续数字化服务。

本轮目标不是简单更换公司名称和配色，而是建立一套能同时传达以下三点的公开门户：

1. **医药行业语境**：页面洁净、克制、严谨，能够让机构客户感到业务边界清楚。
2. **数字技术能力**：通过数据结构、网格、轴线、折射和真实交付物表达技术感，不依赖代码雨、霓虹粒子或虚构仪表盘。
3. **企业可信度**：不虚构客户、效果数字、认证或诊疗能力；所有文案聚焦真实服务范围、实施方式与可验证交付。

首页必须在首屏内回答四个问题：公司是谁、服务谁、解决什么问题、下一步如何沟通。

## 2. 已确认的关键决策

- 公司全称：**北京云医荟科技有限公司**。
- 门户定位：**面向机构客户的医药产业数字化技术服务商**。
- 核心客户：药品生产企业、医疗机构、医药商业公司。
- 核心能力：医药业务软件、数据平台与分析、AI 工作流工具、系统集成与持续服务。
- 视觉方向：用户从三套医药科技方向中选择 **方向 2「医药棱镜 / Pharma Prism」**。
- 首屏主张：**让医药数字化，落到真实业务里。**
- 页面主要为明亮珍珠白与冷银空间；深医药蓝负责文字、结构和深色数据区；折射紫只在棱镜主视觉中少量出现。
- 网站面向机构客户，不提供个人医疗咨询、诊断、治疗或用药建议。
- 不新增外部 UI、动画或字体依赖；使用现有 Next.js、React、HeroUI 和 CSS 能力完成。
- 保留现有线索 API、SQLite 表结构、管理员会话和部署架构，不因视觉重设计执行生产数据库迁移。

## 3. 非目标

本轮不包含以下工作：

- 不改造为患者端、医生端、互联网医疗或在线问诊产品。
- 不宣称临床决策、诊断、治疗、用药推荐或医疗器械软件能力。
- 不编造客户案例、客户 Logo、项目规模、提升百分比、行业排名或合规认证。
- 不上线尚未经过公司确认的资质、证书、备案编号或监管机构标志。
- 不重写管理员后台业务功能，不在本轮处理分页、统计刷新、CSV 注入等后台产品问题；只更新必要的可见品牌文案。
- 不重命名现有数据库文件、Cookie 名称、线索 ID 前缀、Docker 服务、镜像、部署目录或其他内部标识，避免把品牌视觉工作扩大为运维迁移。
- 不新增分析埋点、CRM 同步、邮件通知、短信通知或第三方表单服务。
- 不解决此前项目体检中发现的 Nginx TLS、备份降级、镜像可复现性等运维问题；这些应形成独立任务。

## 4. 品牌与视觉系统

### 4.1 核心概念

“棱镜”表达复杂医药业务经过结构化后变得清晰：

- 输入是角色、流程、数据、接口和规则；
- 中间是业务梳理、权限设计、口径统一、系统建设和人工复核；
- 输出是可理解、可使用、可验收和可持续迭代的数字系统。

棱镜不是药物分子、细胞或诊疗器械，也不承担任何医学效果暗示。页面中的轴线、折面、坐标和卡片标签均围绕真实数字化交付语义。

### 4.2 标志与字标

公开导航使用“图形标志 + 中文字标 + 英文辅助字标”：

- 图形标志为品牌蓝方形，内部使用“云”字和一个斜向折面负形；形状保持简洁，避免像医院十字、药品包装或政府徽章。
- 中文字标：`云医荟科技`。
- 英文辅助字标：`YUNYIHUI TECHNOLOGY`。
- 正式公司主体在页脚、隐私说明和结构化数据中使用全称 `北京云医荟科技有限公司`。

新增 `app/icon.svg` 作为代码原生站点图标；不使用 AI 生成 Logo 图片，不依赖外部图片资源。

### 4.3 色彩令牌

| 令牌 | 色值 | 用途 |
|---|---:|---|
| `--background` | `#F3F6FA` | 页面冷白背景 |
| `--surface` | `#FFFFFF` | 内容表面、卡片和表单 |
| `--foreground` | `#10233F` | 主文字、深色区背景 |
| `--brand` | `#1758D5` | 主按钮、链接、结构轴线 |
| `--brand-soft` | `#DCE7F4` | 次级表面、边界和图表底色 |
| `--brand-light` | `#4E83EC` | 折射高光和数据连接 |
| `--signal` | `#13B7A6` | 少量成功、连接和状态提示 |
| `--prism-violet` | `#7057E8` | 仅用于棱镜折射与 AI 标签，页面面积不超过约 3% |
| `--border` | `#D8E3EF` | 结构线和卡片边框 |
| `--muted` | `#5B7088` | 辅助正文和说明；在 `#F3F6FA` 上约 4.70:1 |

面积比例目标：珍珠白与冷灰约 75%，深蓝与品牌蓝约 22%，青色与折射紫合计不超过约 3%。普通内容依靠实体面板、边框和留白分层；玻璃效果仅限首屏的三个输入/输出标签与商务咨询表单卡。`#13B7A6` 与 `#4E83EC` 只用于非文本图形、边线、状态点或满足大号文字阈值的内容，不承载普通字号白色文字。

### 4.4 字体与排版

- 使用可靠的本地中文字体栈：`Inter, Noto Sans SC, PingFang SC, Microsoft YaHei, sans-serif`。
- 不引入外部 Web Font，避免字体阻塞和中国网络环境下的可用性问题。
- 桌面 H1：`64–76px`，字重约 `700–750`，行高 `1.02–1.08`。
- 移动 H1：`40–44px`。
- 中文标题字距控制在 `-0.01em` 至 `-0.025em`，不沿用现有 `-0.065em` 的过紧字距。
- 正文桌面与移动端均不得小于 `16px`，主要正文行高约 `1.75–1.9`。
- 英文标签、章节编号和数据维度可使用小号大写字母，但它们只作辅助，中文始终是主要信息。

### 4.5 圆角、阴影与结构线

- 小控件圆角：`8px`。
- 卡片圆角：`14–16px`。
- 大型联系区和浏览器式展示容器：`20–24px`。
- 胶囊形状只用于真实状态或短标签，不把普通按钮、标题和导航全部胶囊化。
- 普通卡片不使用大面积悬浮阴影；只有首屏棱镜标签、重点 CTA 和大型视觉容器使用低透明度蓝灰阴影。
- 页面使用 12 栏网格、细轴线和章节编号建立秩序感。

Hero 下方用结构化横条展示可交付内容：

- 验收清单

这些是能力表达，不是认证标志。若具体项目不包含其中某项，实际合同与方案优先。

### 5.4 服务对象

三张卡片分别面向：

1. **药品生产企业**：项目协同、资料与知识管理、商业运营数据、客户与活动流程工具。
2. **医疗机构**：院内非诊疗业务流程、运营数据、台账管理和知识协作工具。
3. **医药商业公司**：客户与渠道协同、经营数据、订单库存信息连接和业务看板。

不增加“健康科技团队”等第四类入口，以免冲淡已确认定位。

### 5.5 解决方案

四类核心能力：

1. **医药业务系统**

   围绕客户、项目、资料、审批和运营等场景，建设角色清晰、流程可追踪的业务系统。

2. **数据平台与分析**

   梳理多源数据、指标口径与权限边界，建设查询、分析、看板与数据服务能力。

3. **AI 工作流工具**

   将 AI 用于资料整理、知识检索、内容辅助和流程自动化，并为关键结果设置人工复核节点。

4. **系统集成与持续服务**

   根据已有系统、接口条件、数据质量和授权范围评估连接方式，并提供上线支持与持续迭代。

卡片采用 7/5、5/7 的非对称 12 栏布局，打破现有连续同构卡片的模板感。移动端全部变为自然高度单列卡片。

### 5.6 数据叙事深色区

标题：

> 把医药数据，变成清晰的行动依据。

流程分三组：

1. **多源业务信息**：现有系统与表格、客户项目与渠道记录、授权知识与内容资料。
2. **统一与治理**：角色权限、指标口径、接口条件、AI 人工复核节点。
3. **可用数字能力**：业务系统、协作工具、专题分析、经营视图和持续迭代能力。

该区使用深医药蓝而非黑色赛博背景。数据链只做一次进入动画，不循环闪烁。

### 5.7 交付方法

四阶段：

1. 场景与边界：确认角色、流程、目标、数据类型和适用要求。
2. 蓝图与原型：形成需求蓝图、交互原型、权限矩阵和阶段范围。
3. 研发与验证：按阶段演示、持续联调并对照验收清单确认。
4. 上线与迭代：完成部署、培训与问题跟踪，按约定持续演进。

桌面为四列连续结构，移动端改为纵向轨道。任何信息都不能依赖 hover 才可见。

### 5.8 场景示意

在没有客户授权和真实项目素材前，统一使用“场景示意”，不用“客户案例”表述：

- 药企业务协同平台；
- 医疗机构非诊疗运营工具；
- 医药商业经营与渠道协同。

示意图只能使用抽象业务字段、结构线和脱敏占位，不出现真实或虚构患者、病历、处方、检查结果和客户数据。

### 5.9 可信设计

展示四个可按项目设计的能力：

- 最小权限；
- 关键操作留痕；
- AI 关键结果人工复核；
- 备份与恢复策略。

用语必须保留条件边界，例如“可按项目需求设计”“具体措施以方案与验收结果为准”。不得出现“绝对安全”“银行级安全”“全面合规”或未经核实的认证。

## 6. 隐私说明

新增公开页面 `/privacy`，内容为一份适用于当前商务咨询表单的基础说明，而不是完整法律合规结论。页面至少明确：

- 处理者：北京云医荟科技有限公司；
- 主动填写信息：咨询方向、单位名称、联系人、联系电话、微信、地区、时间、范围、预算和业务描述；
- 自动记录信息：IP 地址和浏览器 User-Agent，用于防滥用、安全防护与问题排查；
- 处理目的：回复本次商务咨询、评估项目范围、保障表单安全；
- 禁止提交患者、受试者、病历、处方、身份证件和其他敏感个人信息；
- 保存期限：未建立合作的咨询信息原则上在最后一次沟通后 12 个月内删除或匿名化；已建立合作的，按照合同和适用要求处理；
- 不出售咨询信息；因托管、运维或履行项目确需第三方处理时，应根据实际情况另行说明并采取合同与安全措施；
- 权利请求：可通过本网站商务咨询表单选择“隐私与个人信息”联系公司；
- 生效日期：2026-07-10。

该页面及表单文案在生产发布前仍需由公司结合真实联系方式、部署地点、第三方服务和内部数据制度进行法律审阅。此审阅要求是发布门槛，不以页面免责声明替代。

## 7. 组件与代码边界

### 7.1 页面组件

- `components/marketing/marketing-page.tsx`

  继续作为服务端营销页面装配层，负责文案数据、区块顺序和语义 HTML；不负责复杂动画状态。

- `components/marketing/prism-graphic.tsx`

  新增纯展示服务端组件，输出棱镜、轴线和三个语义标签。根节点使用 `aria-hidden="true"`，避免屏幕阅读器朗读装饰内容。

- `components/marketing/reveal.tsx`

  新增小型客户端组件，使用 `IntersectionObserver` 为进入视口的区块添加可见状态。无 Observer 或减少动态效果时直接显示内容，不影响信息访问。

- `components/lead-form/lead-form.tsx`

  保持客户端表单职责，重排必填与选填字段，增加隐私链接、敏感信息提示、网络错误处理和新的提交状态文案。

- `app/privacy/page.tsx`

  新增静态隐私说明页面，复用主视觉令牌，保持内容可读，不使用营销动画。

### 7.2 保持不变的后端边界

- `POST /api/leads` 继续承担 JSON、64KiB、限流、蜜罐、Zod 校验与 SQLite 入库。
- SQLite 表结构和现有字段保持不变。
- 管理员登录、Cookie 会话和后台线索状态流转保持不变。
- 只更新 API 成功提示、管理员页面标题、CSV 表头和可见品牌文案。

### 7.3 元信息与品牌资产

- `app/layout.tsx` 更新 title、description、Open Graph、站点名称与 Organization JSON-LD。
- 新增 `app/icon.svg`，使用代码原生棱镜标志。
- 新增 `app/opengraph-image.tsx`，用 Next.js `ImageResponse` 生成品牌化社交分享图，不依赖外部图片。
- `app/sitemap.ts` 增加 `/privacy`。
- `app/robots.ts` 继续禁止抓取 `/admin` 与 `/api/`，允许公开首页和隐私说明。

## 8. 数据流与错误处理

### 8.1 表单数据流

```text
LeadForm
  -> FormData
  -> 现有字段名 JSON
  -> POST /api/leads
  -> Content-Type / Content-Length / 限流
  -> Zod 校验与蜜罐
  -> createLead
  -> SQLite leads
  -> 返回编号与沟通提示
  -> 客户端展示结果并重置表单
```

### 8.2 错误状态

- 提交期间按钮显示 `正在提交` 并禁止重复提交。
- 415、413、429、400 与 500 继续展示服务端中文消息。
- 网络断开、代理错误或非 JSON 响应统一回退为：`暂时无法提交，请稍后重试或检查网络连接。`
- 异常路径必须在 `finally` 中恢复提交按钮状态。
- 成功后重置字段，但保留屏幕阅读器可读的成功提示。
- 错误提示使用 `role="alert"`；成功与普通状态使用 `aria-live="polite"`。
- 蜜罐命中继续返回不含编号的假成功，避免向机器人暴露拦截逻辑。

### 8.3 敏感信息边界

- 表单不提供文件上传。
- 自由文本旁持续显示“请勿提交患者或其他敏感个人信息”。
- 页面不请求症状、疾病、用药、病历、处方、身份证或医疗记录。
- AI 能力只描述为资料整理、知识检索、内容辅助和流程自动化；关键结果保留人工复核。

## 9. 动效设计

所有动效服务于信息层级，不作为持续背景噪声：

1. Hero 首次加载：棱镜轻微上移与淡入，轴线从原点展开，三个标签依次进入；总时长不超过约 650ms。
2. 章节进入：`Reveal` 只执行一次 `opacity + translateY(8–12px)`。
3. 数据叙事：三个数据块与箭头按顺序点亮一次。
4. 卡片 hover：边框加深并移动最多 2px，不使用大幅上浮。
5. 按钮反馈：140–220ms。
6. 不使用持续旋转棱镜、粒子、扫描线、代码雨、心跳脉冲、鼠标追光、强视差或大面积 blur 动画。
7. 只动画 `transform` 和 `opacity`；避免布局属性动画。
8. `prefers-reduced-motion: reduce` 时取消所有非必要动画并显示最终静态状态。

## 10. 响应式设计

### 桌面（>= 1024px）

- 12 栏布局；Hero 约 7/5；服务卡使用非对称跨栏；数据区为三段横向流程。
- 内容最大宽度约 1180–1240px。

### 平板（768–1023px）

- Hero 保持双栏但减少标题尺寸和视觉占比；服务对象可变为 2+1；解决方案卡变为两列。
- 导航隐藏次要链接，保留 Logo 和 CTA。

### 手机（< 768px）

- 顺序固定为：机构客户标签、H1、副文案、主 CTA、次 CTA、简化棱镜。
- 去掉现有 `min-h-[760px]`；Hero 高度由内容决定。
- 棱镜使用静态、简化多边形，不缩放完整桌面图。
- 服务对象、解决方案与场景全部单列；取消卡片固定最小高度。
- 四阶段交付改为纵向轨道。
- 商务表单单列；选填内容收进原生 `<details>`。
- 触控目标至少 44×44px，正文至少 16px，辅助标签至少 12px。
- 关闭大面积 `backdrop-filter`，保证中低端设备滚动性能。

## 11. 无障碍与性能

- 语义结构使用 `header`、`nav`、`main`、`section`、`article`、`footer`。
- 每个 section 关联可见标题；导航包含清晰的 `aria-label`。
- 装饰棱镜与结构图 `aria-hidden="true"`。
- 所有链接和按钮保留清晰 `focus-visible` 样式。
- 文字与背景达到 WCAG AA 对比度目标；品牌蓝按钮使用白字，浅色辅助蓝不承载小号白字。
- 不以颜色作为唯一状态表达。
- 不引入图片、视频、WebGL、Canvas、动画库或外部字体；首屏主视觉为 CSS/SVG，降低 LCP 与网络依赖。
- 客户端 JavaScript 增量仅限现有表单和一个轻量 `IntersectionObserver` 组件。
- 页面在禁用 JavaScript时仍能展示全部营销内容；只有动态提交与进入动画不可用。
