# 云医荟场景示意升级设计

日期：2026-07-13

## 1. 背景

云医荟首页已经形成“医药棱镜”视觉体系，并在 `main` 分支完成桌面比例、联系表单、响应式和合规可访问性修正。现有“场景实践”区域仍以文字卡片为主，能够说明业务方向，但不足以让访客直观感受到云医荟具备真实业务系统的产品设计与交付能力。

本次设计将场景区升级为三个高完成度的匿名产品原型。视觉可以接近真实项目案例，但不得暗示存在特定客户、虚构认证或未经验证的实施结果。

## 2. 目标

1. 让场景区从“业务说明卡片”升级为“可感知的系统能力展示”。
2. 保持医药棱镜的理性、洁净、精密风格，不改成通用 SaaS 产品商城。
3. 使用匿名界面、合理状态和业务字段提升真实感。
4. 在桌面端形成一主两辅的非对称视觉结构，在移动端保持清晰单栏阅读。
5. 保留统一合规说明，不出现客户名称、客户 Logo、认证、成交金额或效果百分比。

## 3. 方案比较

### 方案 A：保留现有抽象文字卡片

优点：实现成本最低，内容安全。

缺点：产品感和可信度不足，无法形成强视觉记忆。

### 方案 B：匿名真实感产品原型（采用）

使用模拟业务界面、状态、列表、趋势和流程节点，表现典型医药数字化场景。明确标注为场景示意，不映射特定客户。

优点：视觉效果强，能够展示业务理解和产品设计能力，同时保持真实边界。

缺点：需要为桌面和移动端分别设计界面密度。

### 方案 C：包装为具体客户案例

优点：表面上具有更强的社会证明。

缺点：在没有真实授权和证据时会形成误导，损害品牌可信度，因此不采用。

## 4. 信息架构

场景区保留三个主题：

1. **药企业务协同平台**
   - 客户与项目状态
   - 资料完成度
   - 审批与待办
   - 最近跟进

2. **医疗机构运营台账**
   - 非诊疗事项台账
   - 到期提醒
   - 知识协作
   - 部门与责任人状态

3. **医药商业经营分析**
   - 渠道与客户结构
   - 订单和库存概览
   - 经营趋势
   - 异常与重点关注项

## 5. 视觉设计

### 5.1 桌面布局

采用 `1.25fr / 0.75fr` 的两列结构：

- 左侧为“药企业务协同平台”主场景，内部再分为文案区与产品界面区。
- 右侧上下排列“医疗机构运营台账”和“医药商业经营分析”。
- 主场景高度约 340–380px；右侧两个辅助场景平分高度。

### 5.2 主场景界面

主场景右侧展示匿名业务工作台：

- 顶部状态摘要：进行中项目、待审批事项、资料完成度。
- 中部项目列表：使用“项目 A / 项目 B”等匿名名称，展示状态标签和进度条。
- 右下角叠放一张轻量跟进卡或审批卡。
- 不使用真实公司名、人名、患者信息、药品名称或具体销售数据。

界面采用珍珠白、冷银边框和品牌蓝状态，玻璃效果仅用于叠层小卡，不扩大到整个区域。

### 5.3 辅助场景界面

“医疗机构运营台账”使用紧凑台账列表与提醒状态；“医药商业经营分析”使用小型趋势线、渠道分布条和库存状态。

两张卡都保留大号淡色编号，避免信息过多导致小卡不可读。

### 5.4 动效

- 场景区进入视口时沿用现有 `Reveal`，只执行一次。
- 主场景中的状态条允许一次性宽度展开。
- 禁止持续滚动、循环闪烁、自动轮播和鼠标追光。
- `prefers-reduced-motion` 下直接显示最终状态。

## 6. 组件边界

新增独立展示组件，避免继续扩大 `marketing-page.tsx`：

- `components/marketing/scenario-showcase.tsx`
  - 负责场景区整体布局和语义结构。
- `components/marketing/scenario-dashboard.tsx`
  - 负责主场景匿名工作台。
- `components/marketing/scenario-ledger.tsx`
  - 负责医疗机构运营台账。
- `components/marketing/scenario-analysis.tsx`
  - 负责医药商业经营分析。

场景文案继续从 `components/marketing/content.ts` 提供；界面中的模拟展示数据使用只读常量，放在场景组件旁或单独的 `scenario-content.ts` 中，不接入 API 或数据库。

## 7. 响应式规则

### 桌面端（≥1024px）

- 一主两辅布局。
- 主场景内文案和模拟界面并排。
- 关键文字和状态在 1440、1280、1024 宽度均不得截断。

### 平板端（768–1023px）

- 三个场景改为单列。
- 主场景仍可在足够宽度下保留两栏，窄平板改为上下布局。

### 移动端（≤767px）

- 文案在上、界面在下。
- 隐藏低价值辅助列，只保留 3–4 个最重要状态。
- 不产生横向滚动。
- 正文不小于 16px，状态文字不小于 12px，触控目标不小于 44px。

## 8. 可访问性与内容边界

1. 模拟界面属于信息性内容，不能全部标记为 `aria-hidden`；核心状态需要有可读文本。
2. 装饰性趋势线、网格和几何背景使用 `aria-hidden="true"`。
3. 不仅依赖颜色表达状态，状态标签同时提供文字。
4. 保留 `#5B7088` 辅助正文色，不回退到对比度较低的原设计色。
5. 不出现具体客户、医院、药企、患者、受试者、药品或处方信息。
6. 不出现未经验证的效果数字、市场排名或合规认证。

## 9. 测试与验收

### 自动测试

- 内容契约测试确认三个场景标题和统一免责声明存在。
- 结构测试确认场景区使用独立组件，不回退成简单循环文字卡。
- 测试源代码中不存在客户 Logo、知名客户名称、百分比效果承诺或患者信息。
- TypeScript、全量测试和生产构建通过。

### 视觉验收

至少检查：

- 1440×1000
- 1280×800
- 1024×768
- 768×1024
- 390×844
- 375×667
- reduced-motion

重点检查：

- 主场景产品界面没有文字溢出。
- 右侧两张卡高度和视觉重量平衡。
- 移动端界面简化后仍能理解三个场景。
- 页面整体长度没有因模拟界面显著失控。
- 场景区与前后的交付方法、Trust 区保持清晰节奏。

## 10. 非目标

- 不开发真实业务系统或可交互后台。
- 不新增咨询 API、数据库表或管理后台字段。
- 不创建虚构客户详情页。
- 不展示未经授权的 Logo、品牌、人物头像或医疗数据。
- 不重做首页其他章节和品牌 Logo。

## 11. 完成标准

访客在不阅读长段文字的情况下，能够从三个场景界面迅速理解云医荟可交付：业务协同系统、运营台账工具和经营分析平台；同时页面明确表明这些是行业场景示意，而非真实客户案例。
