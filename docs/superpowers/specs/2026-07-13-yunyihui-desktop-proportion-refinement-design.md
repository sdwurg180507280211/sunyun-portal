# 云医荟门户桌面比例与纵向节奏优化设计

日期：2026-07-13

状态：方向已获授权，等待书面规格复核

基础设计：[2026-07-10-yunyihui-pharma-prism-design.md](./2026-07-10-yunyihui-pharma-prism-design.md)

## 1. 背景与问题

当前正式首页在 1440px 桌面视口使用固定 `1200px` 内容容器，在 1920px 视口仍保持 `1200px`。它在常见笔记本上尚可，但在 1600–1920px 显示器上两侧空白快速扩大，卡片、交付结构和联系面板都显得偏窄。与此同时，Hero 最低高度为 `700px`，普通章节上下留白多为 `96px`，多类卡片在手机和桌面都保留固定最小高度，形成“横向偏窄、纵向偏长”的整体比例。

用户选中的 `width: min(1680px, 84%)` 只作为“桌面需要更舒展”的信号，不直接采用：在 1440px 视口下它只有约 `1210px`，与现状几乎相同；在更宽视口下又会让正文、信任信息和表单不受区分地一起拉宽。

本轮目标是建立明确的宽度层级，让结构型内容在桌面上更舒展，同时保持医药 B2B 门户应有的克制、可读性和精密感。

## 2. 方案比较与决策

### 方案 A：分层容器（采用）

建立“宽展示容器、标准内容容器、阅读宽度”三层体系。卡片网格、流程结构和大型联系面板使用宽展示容器；可信说明等低密度结构使用标准容器；标题和正文继续限制行宽。

优点：能直接缓解宽屏的狭窄感，又不会把所有文字和表单机械拉长；适合当前页面既有的 12 栏结构。代价：需要显式标记各章节使用哪一层容器，并补充视觉契约测试。

### 方案 B：统一扩大为 1440px（不采用）

所有 `.site-shell` 一次性扩大。实现最简单，但正文、可信设计、页脚说明与表单也会同步变松，页面缺少宽窄层次。

### 方案 C：`84% / 1680px` 全流体（不采用）

接近设计草稿的百分比布局，但 1440px 下改善很小，超宽屏下又过度扩张，且不同区块无法根据内容密度选择合适行宽。

## 3. 容器体系

### 3.1 宽展示容器

- 最大宽度：`1440px`。
- 适用区域：顶部导航、Hero 内部网格、交付物横条、服务对象、解决方案、数据与交付结构、场景示意、商务咨询面板和页脚。
- 用途：承载多列卡片、横向流程、品牌主视觉与大型组合面板。

### 3.2 标准内容容器

- 最大宽度：`1280px`。
- 适用区域：可信设计等需要更紧密阅读关系、但又不应退回 `1200px` 的内容结构。
- 后续章节若没有明确指定，默认采用标准内容容器，不自动使用宽展示容器。

### 3.3 阅读宽度

- 章节标题最大宽度：`840px`。
- 普通说明正文最大宽度：`720px`。
- Hero 副文案最大宽度：保持 `672px`。
- 容器变宽不改变正文最小字号、行高和 WCAG AA 对比度要求。

### 3.4 响应式页边距

两类页面容器共享同一组流体页边距，总水平留白使用 `clamp(32px, 6.4vw, 112px)`；容器实际宽度为“最大宽度”和“视口扣除总页边距后的宽度”两者中的较小值。

预期结果：

| 视口 | 宽展示容器 | 标准内容容器 | 说明 |
|---:|---:|---:|---|
| `1280px` | 约 `1198px` | 约 `1198px` | 保留约 41px 单侧边距 |
| `1440px` | 约 `1348px` | `1280px` | 展示区明显舒展，阅读区仍收敛 |
| `1920px` | `1440px` | `1280px` | 宽屏不再固定在 1200px，也不无限扩张 |

不以固定百分比作为唯一宽度规则；超宽屏继续受最大宽度约束。

## 4. 页面应用规则

### 4.1 Hero 与导航

- Hero 背景继续全宽铺开，7/5 内容网格放入宽展示容器，与导航和后续结构区左边界对齐。
- 桌面 Hero 最低高度从当前 `700px` 收紧到 `660px`；窄屏高度由内容决定，不强行压缩棱镜或文字。
- H1 仍保持两行品牌主张和既有阅读宽度，不因容器加宽而继续放大。
- 棱镜尺寸可利用新增横向空间，但不得突破容器、遮挡文字或产生水平溢出。

### 4.2 章节与卡片

- 普通章节纵向留白由统一 `96px` 调整为手机约 `64px`、桌面约 `80px`。
- 服务对象章节恢复白底，解决方案保持珍珠灰底，重新建立白/灰章节节奏。
- 服务对象卡片在桌面保留约 `224px` 的结构高度；解决方案卡片约 `208px`；主场景卡约 `288px`。这些是桌面层级，不下放为手机固定最小高度。
- 手机端卡片全部由内容决定高度，避免大面积空白；平板和桌面仅在同一行需要对齐时使用最小高度。
- 标题与正文行宽继续受阅读宽度约束，新增横向空间优先分配给列间距、卡片内容和结构图，而不是延长文字行。

### 4.3 联系区与页脚

- 商务咨询面板使用宽展示容器，但保留已确认的 `.78fr / 1.22fr` 分栏、真实字段、选填折叠、隐私同意、敏感信息警示、校验和提交状态。
- 表单字段自身不使用超长单列；桌面继续按现有两列字段网格组织，业务场景、同意项和提交按钮保持跨列。
- 页脚使用宽展示容器以对齐页面边界，但说明文字继续限制最大宽度并保持可读行长。

## 5. 范围边界

本轮只处理宽度层级、Hero 与章节纵向比例、卡片高度策略和服务对象背景节奏，不同时扩大为完整的草稿还原任务。

明确不包含：

- 不改商务表单字段、API、SQLite、隐私说明或后台逻辑。
- 不拆分当前合并的数据区与交付流程。
- 不新增场景 01 仪表盘、浮层数据卡或客户案例。
- 不重写棱镜图形、营销文案、章节编号或动效系统。
- 不新增 npm 依赖、外部图片、字体或分析脚本。
- 不修改 `.superpowers/` 与 `graphify-out/`。
- 不处理生成文件 `next-env.d.ts` 的现有本地差异，也不把它纳入本轮提交。

上述草稿还原差异可在桌面比例优化完成后单独评估，避免宽度调整与内容重构互相干扰。

## 6. 实现边界

- 在 `app/globals.css` 中定义共享页边距、标准容器和宽展示容器；不引入运行时布局计算。
- 在 `components/marketing/marketing-page.tsx` 中为各章节选择明确的容器层级，并调整必要的响应式高度类。
- 保留 `.section-title`、`.section-copy` 等阅读宽度约束，必要时只做小幅上限校正。
- 以 `tests/visual-contract.test.ts` 记录宽度层级、关键章节容器选择、自然高度移动卡片和联系区分栏不变等契约。
- 所有修改保持服务端渲染和渐进增强，不增加客户端 JavaScript。

## 7. 验证与验收

### 7.1 自动验证

- 先增加会失败的视觉契约测试，再实现布局修改。
- 运行 `npm test`、`npm run typecheck` 和 `npm run build`。
- 确认没有引入依赖、API、数据库或公开文案变化。

### 7.2 浏览器验证

验证视口：`1920×1080`、`1440×1000`、`1280×800`、`1024×768`、`768×1024`、`390×844`。

每个视口检查：

- 页面无水平溢出、文字遮挡或棱镜裁切。
- 1440px 与 1920px 下卡片和结构区明显宽于旧版 `1200px` 容器。
- 标题和正文没有因容器扩大形成过长行宽。
- 1440px 下整体纵向长度比旧版更紧凑，但章节仍有清楚停顿，不出现内容拥挤。
- 1024px 附近布局切换稳定；导航和 CTA 不碰撞。
- 390px 下卡片为自然高度，表单、隐私同意和触控目标保持可用。
- 联系区的真实提交、错误恢复和键盘路径不回归。
- `prefers-reduced-motion: reduce` 下内容直接可见。

### 7.3 完成标准

1. 宽展示容器、标准内容容器和阅读宽度职责清楚，不再依赖单一 `1200px` 容器。
2. 正式页在 1600–1920px 桌面上不再呈现明显的中央窄条感。
3. 1440px 常见桌面上展示区获得可感知的横向提升，正文仍保持舒适行长。
4. 页面纵向节奏更精炼，移动端不再继承桌面卡片固定高度。
5. 服务对象与解决方案恢复明确的白/灰章节分隔。
6. 商务咨询、隐私、API、数据库、后台和品牌合规能力保持不变。
7. 自动测试、类型检查、生产构建和六个目标视口的浏览器检查全部通过。
